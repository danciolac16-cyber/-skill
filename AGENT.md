# 智能体书籍知识库 Agent Guide

本文件是 `F:\智能体书籍知识库` 的项目级工作规则。后续 Codex 在本项目中处理书籍、Kimi/DeepSeek/Gemini/Claude 等外部模型稿、正式蒸馏、索引、PDF/HTML、GitHub 同步时，必须优先遵守本文件。

## 0. 核心目标

本项目不是普通读书笔记项目，而是为 AIGC 智能体建立可调用的外部知识库。

最终产物必须让后续智能体能够：

- 读取模块职责，而不是只看到摘要。
- 根据触发条件自动选择模块。
- 使用输入字段、控制参数、工作流、输出 Schema、Prompt 模板、质量检查器、负向约束、修复规则执行任务。
- 在不同模块之间建立清晰分工和调用链。
- 通过临时索引、质量评级、旧版替换记录追踪当前版本是否可用。

禁止把“好看的总结”当成正式蒸馏。没有执行层、没有评级、没有来源披露、没有质量检查器的内容，原则上不能进入最终可用库。

### 0.1 AIGC 视频生产总目标

本知识库的最终服务对象不是泛用聊天智能体，而是 **AIGC 视频生产智能体系统**。主要任务包括：

- AI 短剧、影视、小说视觉化、广告片、概念片、预告片、品牌内容和短视频。
- 策划、研究、世界观、剧本、人物、对白、分镜、镜头、调度、表演、声音、美术、布光、色彩、剪辑、营销和质量控制。
- 从用户意图到生产任务书、生成 Prompt、素材生成、镜头连续性检查、成片诊断和失败修复的完整管线。

每本书都必须回答：它在 AIGC 视频生产管线中负责哪一个决策，向上游索取什么，向下游交付什么，能够控制哪些可观察的视听结果。

正式模块至少要声明：

```json
{
  "aigc_video_role": "core | support | reference | not_directly_applicable",
  "production_stages": [],
  "supported_content_types": [],
  "upstream_inputs": [],
  "video_control_parameters": {},
  "downstream_handoff": [],
  "generation_prompt_hooks": [],
  "video_quality_checks": [],
  "video_repair_rules": [],
  "cross_shot_consistency_rules": []
}
```

适用生产阶段统一使用：

- `development`：选题、研究、概念、受众、品牌和项目定位。
- `writing`：故事、剧本、人物、对白、世界观和节拍。
- `preproduction`：视觉开发、分镜、镜头表、美术、服化道、布光和声音设计。
- `generation`：图像、视频、角色、场景、动作、对白、声音和音乐生成。
- `postproduction`：剪辑、节奏、调色、声音、字幕、合成和交付。
- `distribution`：广告版本、平台适配、封面、标题、传播和转化。
- `quality_control`：连续性、角色一致性、空间逻辑、视听意图、品牌一致性和技术质量。

执行约束：

1. 不得只写“可用于 AI 视频”。必须给出具体生产阶段、任务、输入、参数、输出和检查器。
2. 理论书也要转成生产决策或诊断规则；若确实不能直接控制视频，应标记 `not_directly_applicable` 并只作为上游参考，不强行视频化。
3. 小说和文学模块不能只存主题与人物简介，至少要提供改编单位、场景资产、人物状态、视觉母题、叙事约束和不可破坏的原作边界。
4. 影视制作类模块应优先输出镜头、景别、机位、焦距、调度、动作、空间、光色、声音、剪辑和连续性参数。
5. 广告营销类模块应连接品牌人格、受众欲望、传播目标、脚本结构、视听 Hook、CTA 和版本测试，但不得越权替代摄影、剪辑等专业模块。
6. 美学与风格类模块必须把抽象风格转成可观察参数，并检查“风格是否服务叙事”，不能只产出形容词堆。
7. Prompt 只是接口之一，不是最终能力证明；必须同时具备结构化输出、质量检查和失败修复。
8. 单镜头漂亮不等于视频可用。涉及生成时必须检查角色、服装、道具、空间、光向、色彩、动作、视线、时间和声音的跨镜头一致性。
9. 现代模型、平台、软件和合规规则属于 `modern_supplement`，不得冒充原书知识。
10. 如果模块不能明确改善 AIGC 视频生产中的决策、生成、检查或修复，不能仅凭书籍总结完整而评为 A。

## 1. 固定目录结构

项目根目录应保持干净，只保留固定入口：

- `AGENT.md`：本文件，项目级规则。
- `目录说明.md`：面向人工查看的目录说明。
- `html\`：正式 HTML 归档、待升级归档、旧版归档、补强附件。
- `PDF\`：正式 PDF 归档、待升级归档、旧版归档、补强附件。
- `书籍资料\`：源书、EPUB/PDF/DOCX/MOBI、小说资料和专题资料。
- `kimi初步蒸馏\`：外部模型初稿、Codex 守门报告、补强报告、原料包。
- `临时索引_阶段整理\`：清单、评级、移动记录、旧版替换记录、OCR/全文抽取样本、守门散件。
- `工具\`：脚本、辅助工具、OCR/转换工具等。

根目录不要堆放书籍、蒸馏稿、HTML、PDF、CSV 或临时文件。

源书放入 `书籍资料\` 的主题文件夹：

- `01_影视制作_导演摄影剪辑美术声音`
- `02_编剧叙事_故事结构人物对白`
- `03_广告营销_媒介传播转化`
- `04_色彩光线_美学艺术风格`
- `05_视觉设计_动效字体排版`
- `06_互动游戏叙事`
- `07_故事形态学专题资料`
- `08_待归类_原始文件`
- `09_理论补充_空间媒介社会学`
- `小说资料\...`

无法判断类别时，先放 `08_待归类_原始文件`，不要随意塞进根目录。

### 2.2 Kimi 或外部模型稿

外部模型稿统一放入 `kimi初步蒸馏\`：

- `01_外部模型初稿`：Kimi/DeepSeek/Gemini/Claude 等外部模型第一步分析、蒸馏蓝图、供 Codex 审核稿、交接包、未正式归档稿。
- `02_Codex守门报告`：Codex 守门审阅结论、可用性判断。
- `03_补强报告与原料包`：A-/A 补强报告、BlackBox 原料包、公式测试、升级说明。

不要把外部模型稿直接当正式归档。

归类时必须优先看内容，不只看文件名：

- 标题或正文写着“第一步分析”“供 Codex 审核/审阅”“待 Codex 守门”“Kimi 交接”等，通常仍属于外部模型初稿，优先放 `01_外部模型初稿`。
- 只有正文明确包含 Codex 的守门结论、可用性判断、评级上限、补强路径、是否可直接归档等内容时，才放 `02_Codex守门报告`。
- 如果文件名像“守门报告”，但正文其实是外部模型自评或请求 Codex 审核，不能放入 `02_Codex守门报告`。
- 如果无法判断，先打开前 50-100 行阅读，再移动；不要按关键词机械归类。

当前知识库仍在建设期，不激进删除外部模型初稿、交接稿、补强稿；不确定是否有用的旧件优先隔离到 `临时索引_阶段整理\06_旧件隔离_待确认`，而不是直接删除。

### 2.3 HTML/PDF 正式文件

HTML 放 `html\`，PDF 放 `PDF\`，并按状态分类：

- `正式归档_已升级`：已通过守门，可供后续智能体调用。
- `正式归档_待升级`：已有文件，但质量不足或仍需重蒸。
- `旧版_已替换`：被新版替代但暂时保留追溯。
- `补强附件_守门报告`：补强报告、审计报告、评分说明、附录。

同一书名不要在多个状态文件夹中留下未说明的重复版本。若新旧版都保留，必须在旧版替换记录中写清楚关系。

### 2.4 临时索引与记录

所有清单、评级、替换记录、移动记录放入 `临时索引_阶段整理\`：

- `01_质量体检与分级`：ABCD 评级表、质量审计。
- `02_清单与索引`：全量补强清单、临时索引、阶段摘要。
- `03_旧版替换记录`：旧版删除/备份/替换关系、移动记录。
- `04_全文抽取与OCR样本`：全文抽取、OCR 样本、页图审计。
- `05_守门员审计散件`：散落的守门结论、未合并审计片段。
- `06_旧件隔离_待确认`：乱码旧审计、失效抽取样本、疑似重复但暂不确定能否删除的材料；此目录不作为主索引读取。

轻量 Manifest 固定放在 `02_清单与索引`：

- `模块调用总表_agent_manifest.csv`
- `模块调用总表_agent_manifest.json`

每完成一本正式归档，都必须追加或更新 Manifest。Manifest 当前是施工索引，只收录 `html\正式归档_已升级` 与 `PDF\正式归档_已升级` 成对模块，用于后续 SOP 智能体按领域、任务、触发关键词、上下游关系定位模块。

Manifest 准入以“重蒸/正式升级证据”为准，不以历史质量表评级单独决定：

- 可进入 Manifest：已在 `html\正式归档_已升级` 与 `PDF\正式归档_已升级` 成对存在，或守门/补强记录明确写明已重蒸、已正式归档、冲 A 成功、进入正式归档，并且成品已放入正式升级区。
- 不可进入 Manifest：仅质量表写 A/A-、仍在 `正式归档_待升级`、只有 Kimi 初稿/交接包/BlackBox 原料包、只有旧版摘要或用户曾经提过但未生成正式升级版。
- `已重蒸模块登记表.csv` 与 `重复重蒸拦截索引.csv` 是防止漏收和重复重蒸的辅助事实源；后续用户再次提出同一本书时，必须先查这些表和 Manifest。

重复重蒸拦截规则：

1. 用户提出某本书要重蒸时，先查 `已重蒸模块登记表.csv`、`模块调用总表_agent_manifest.csv`、正式升级 HTML/PDF 和旧版替换记录。
2. 若已重蒸，先告知已重蒸模块名、当前评级、已完成步骤（HTML/PDF、Manifest、守门/补强、旧版处理），并建议走“补强升级”而不是重新蒸馏。
3. 默认不直接二次重蒸。若用户明确坚持二次重蒸，必须作为版本升级处理：旧版进入 `旧版_已替换`，写替换记录，Manifest 只保留新版入口。

## 3. 标准工作流总览

每次开始处理一本书、一个 Kimi 初稿或一个归档模块前，先判断任务类型，再选择对应流程。不要跳过查重、来源审计和用户确认。

### 3.1 前置需知：先判断任务类型

常见任务类型：

- 新书第一步分析：用户刚给出书名或源书，尚未正式蒸馏。
- Kimi/外部模型初稿守门：用户给出 Kimi、DeepSeek、Gemini、Claude 等模型的蒸馏稿或交接稿。
- Codex 替代 Kimi 做初稿：书籍超过外部模型限制，或用户要求 Codex 直接做第一步分析。
- 正式蒸馏归档：用户确认进入正式 HTML/PDF 归档。
- 已归档模块补强：已有正式模块，但需要补图像审计、公式测试、参数化、资质链证据或 A 级补强。
- 文件夹/Manifest 审计：检查错放、漏收、断链、重复、目录污染。
- 重复重蒸检查：用户再次提出已处理过的书。

每次开始前必须先查重：

1. `临时索引_阶段整理\02_清单与索引\已重蒸模块登记表.csv`
2. `临时索引_阶段整理\02_清单与索引\重复重蒸拦截索引.csv`
3. `临时索引_阶段整理\02_清单与索引\模块调用总表_agent_manifest.csv`
4. `html\正式归档_已升级` 与 `PDF\正式归档_已升级`
5. `html\旧版_已替换` 与 `PDF\旧版_已替换`

如果已重蒸，先提醒用户当前评级、已完成步骤和可选的补强升级路径，不直接重新蒸馏。如果源书或 Kimi 初稿缺失，先说明缺什么，不假装已读。

### 3.2 拿到 Kimi/外部模型初稿时

处理顺序：

1. 定位源书：在 `书籍资料\` 中找 EPUB/PDF/DOCX/MOBI；找不到则说明缺源书。
2. 定位外部稿：优先在 `kimi初步蒸馏\01_外部模型初稿` 中找对应初稿、交接包或供审稿。
3. 内容归类：打开正文判断它是外部初稿、Codex 守门报告还是补强原料包；不要只看文件名。
4. 来源审计：确认格式、大小、是否加密、是否可抽取、是否扫描、是否需要 OCR、是否有教学图像。
5. Codex 自读原书：只做目录、字符数、关键词、抽样页、contact sheet 或外部模型自述，不能称为正式守门；可读原书必须由 Codex 完成全文/全结构单元处理，扫描或图像型书籍必须完成可说明覆盖范围的 OCR/图像审计。
6. 双向核对外部稿：必须同时做“原书 → 外部稿”和“外部稿 → 原书”两条核对链：
   - 原书 → 外部稿：检查原书重要章节、图像、案例、概念、警告、方法是否被外部稿遗漏、弱化或误解。
   - 外部稿 → 原书：检查外部稿中的断言、案例、视觉描述、现代补强、Prompt、评级、规则是否有本地证据，是否错误归因、过度泛化或与原书冲突。
7. 图像/图表守门：图像型书籍不能只看文字。Codex 需要直接检查页面缩略图、抽取图像或代表性渲染页；若无法逐图看完，必须披露覆盖范围并降低评级。
8. 反伪蒸馏检查：检查是否有触发条件、输入字段、控制参数、工作流、输出 Schema、Prompt 模板、质检器、修复规则。
9. 评级判断：给当前评级、可冲评级、硬上限原因和是否能供后期智能体调用。
10. 守门结论：明确“可直接归档 / 需要 Codex 补强 / 需要 Kimi 重蒸 / 暂不可用”。
11. 用户确认：除非用户明确要求直接做，否则先输出守门结论，确认后再正式生成 HTML/PDF。

硬性命名规则：

- 只完成文件验证、抽取统计、目录/关键词核验、图像 contact sheet 或结构预览时，只能叫 `守门预审`、`pre_gate_audit` 或 `initial_guard_review`。
- 只有完成 Codex 自读原书 + 双向核对矩阵后，才能叫 `正式守门` 或 `final_gatekeeping`。
- 没有双向核对矩阵时，最高评级为 A-，不得评 A，不得直接进入正式归档；除非用户明确接受临时非最终模块。
- 若发现外部稿存在多项重大无来源断言、现代补强冒充原书、图像描述虚构或原书重要机制漏掉，最高 B，修复后再重评。

文件归类：

- Kimi 初稿、交接包、供审稿保留在 `01_外部模型初稿`。
- Codex 守门结论放 `02_Codex守门报告`。
- BlackBox、补强报告、升级报告、公式测试放 `03_补强报告与原料包`。

### 3.3 Codex 守门输出必须包含

守门输出至少包含：

- 守门结论：是否可用、当前评级、可冲评级、是否可进入正式归档。
- 本地证据：源文件路径、抽取状态、阅读覆盖、OCR/图像审计状态。
- 外部稿风险：是否伪蒸馏、是否漏执行层、是否声称全文但证据不足。
- 双向核对矩阵：原书有什么被外部稿漏掉；外部稿写了什么但原书证据不足；每项标注严重度与修复动作。
- 缺口清单：Source Audit、Reading Coverage、Agent Capability Delta、Trigger/Input/Workflow/Output、Quality Checker、Repair Rules、BlackBox JSON 等是否缺失。
- 决策：直接归档、补强后归档、退回重蒸或仅保留为参考稿。

### 3.4 正式蒸馏归档流程

正式归档必须按以下顺序执行：

1. 生成正式 HTML。
2. 生成正式 PDF。
3. 放入 `html\正式归档_已升级` 与 `PDF\正式归档_已升级`。
4. 执行第二步自检：HTML/PDF 是否成对、文件是否可打开、评级是否标准、是否包含执行层、是否有 Manifest 条目。
5. 更新索引：Manifest CSV/JSON、已重蒸模块登记表、重复重蒸拦截索引、质量分级表、临时索引、全量重蒸补强清单。
6. 如替换旧版：旧版移入 `旧版_已替换`，写旧版替换记录。
7. 如只是补强：保留同一主模块入口，写补强记录，更新登记表和 Manifest notes。

每次正式归档后必须更新：

- `模块调用总表_agent_manifest.csv`
- `模块调用总表_agent_manifest.json`
- `已重蒸模块登记表.csv`
- `重复重蒸拦截索引.csv`

视情况更新：

- `已归档质量分级_清理后.csv`
- `已蒸馏书单_临时索引.csv/md`
- `全量重蒸补强清单.csv/md`
- `旧版替换记录/*.csv`
- `移动记录/*.csv`
- `post_generation_self_check_*.json`

审计类任务常见输出：

- `待用户确认修复清单.csv`
- `目录污染清单.csv`
- `kimi目录内容分类审计.csv`
- `SOP调度验证_日期.csv`

### 3.5 资质链检查流程

每个已重蒸模块需要检查：

- HTML 正式升级版。
- PDF 正式升级版。
- Manifest 条目。
- 标准评级。
- Source Audit。
- Reading Coverage。
- Agent Capability Delta。
- Executable Modules。
- JSON / Prompt / Quality Checker / Repair Rules。

资质链状态分为：

- `完整`：可直接给后续智能体调用。
- `可用但需补证据`：位置与调度可用，但缺少部分审计/披露/JSON/质检显式标记。
- `位置正确但内容薄`：已在正式区，但需回查是否真的达到新标准。
- `疑似误收/漏同步`：位置、Manifest 或路径存在硬问题。
- `未重蒸不处理`：不进入 Manifest，也不作为修复问题。

### 3.5.1 最终自检与批量归档合格门槛

当用户要求“最终自检”“全量复查”“机器一致性”“75 本/批量归档是否全部对齐”时，不能只做字段扫描，也不能用脚本命中字段名替代正文阅读。必须把正式归档模块逐个打开检查，并输出可追责的审计结果。

批量最终自检的最低检查顺序：

1. 总账完整性：核对主清单数量，确认每条正式模块都有审计记录。
2. 问题清单归零：检查 P0/P1/P2 是否仍有残留；残留问题必须列模块名、问题类型和修复建议。
3. 文件资质链：逐条验证 Kimi 初稿、Codex 守门/补强记录、正式 HTML、正式 PDF、PDF 文本抽取、标题一致性、SHA/路径记录和旧版隔离/替换记录。
4. 索引同步链：Manifest CSV/JSON、已重蒸登记表、重复重蒸拦截索引、质量分级表必须唯一命中，不得有重复主入口或路径断链。
5. 正文颗粒度与执行层：逐个查看正式 HTML 正文，不只看目录和机器锚点。必须判断 Source Audit、Trigger、Inputs、Controls、Workflow、Output Schema、Prompt、QC、Negative Constraints、Repair、Cross-Module Dispatch、Agent Capability Delta、Black-Box JSON 是否有实质内容。
6. 知识分层与现代适配：检查 `original_book`、`modern_supplement`、`related_module`、`unverified_external`、`unavailable_source` 是否在正文中真正使用；现代补强不得提高原书证据评级。
7. AIGC 转化层：按模块职责检查 AIGC 接口，不强行全链路，但必须说明生产阶段、输入、可观察控制参数、handoff、prompt hooks、video QC、repair 和跨镜头一致性。
8. 外部模型守门层：凡使用 Kimi/Gemini/Claude/DeepSeek 等外部稿，必须检查是否有 Codex 本地证据优先与双向核对记录；外部稿只能作为 draft、benchmark 或 clue。
9. BFI/导演风格专项：影视作者、流派和导演风格不得被“一刀切”删除。若涉及在世导演或风格风险，采用“保留创作机制与可观察参数、标注来源层、避免冒充原作或复制具体受保护表达”的分层规则，而不是砍掉风格能力。

正文密度状态统一使用：

- `substantive_dense`：字段完整且内容专属、可执行、可检查、可修复。
- `substantive_adequate`：字段完整，正文可用，但少量参数可后续增强。
- `substantive_but_thin`：有正文，不是空锚点，但颗粒度低于同批标准。
- `machine_aligned_or_thin`：字段名齐全但主要是机器锚点、模板句或泛泛补丁。

最终合格状态统一使用：

- `qualified`：文件、索引、正文、分层、AIGC 接口和守门链全部过关。
- `needs_density_review`：主链路成立，但正文细度不足。
- `needs_minor_repair`：轻微命名、备注、索引或格式问题。
- `needs_gatekeeping_repair`：缺 Codex 双向守门、本地证据不足或外部模型混源。
- `needs_bfi_style_repair`：导演风格、类型手法或创作风险被一刀切删除，需要分层恢复。
- `not_qualified`：路径断链、正式归档不成立、严重混源或 HTML/PDF 不可用。

只有满足以下条件时，才能向用户说“批量正式归档已经全部对齐”：

- 主清单每条都有审计记录。
- `qualified` 覆盖全部正式模块，或残留项已被明确标为不参与本轮正式归档。
- `machine_aligned_or_thin` 为 0。
- P0/P1/P2 问题清单为 0，或报告中明确列出尚未修复的问题。
- HTML/PDF/Manifest/登记表/重复拦截索引/质量表缺口为 0。
- 现代适配层没有污染原书证据层。
- AIGC 接口达到该模块职责对应的最低等级：核心生产模块 V3，理论/参考模块 V2 或 `V3_when_routed`，不直接生产模块必须说明 `not_directly_applicable` 和路由职责。

### 3.5.2 标准版本、守门报告与问题队列 Schema

为避免长期批量归档后无法判断“某模块按哪一版规则生成”，正式模块、守门报告、补强记录和批量审计表应尽量携带规则版本与统一问题字段。旧模块不因缺少这些字段自动作废，但进入后续补强或批量复查时应补齐。

#### A. 标准版本字段

每个正式模块或 post self-check 建议写入：

```json
{
  "standard_version": {
    "agent_md_commit": "",
    "skill_commit": "",
    "ruleset_name": "agent_book_distillation",
    "ruleset_version_label": "",
    "generated_under_rules_date": "",
    "requires_reaudit_if_rules_after": [],
    "notes": ""
  }
}
```

使用规则：

- `agent_md_commit`：项目 `AGENT.md` 对应的 Git commit 或本地哈希；无 Git 时写文件 SHA。
- `skill_commit`：`agent-book-distillation` 对应的 Git commit 或本地哈希。
- `ruleset_version_label` 可使用人工标签，例如 `kimi75_final_self_check_v1`、`positive_flywheel_v1`。
- 规则版本字段只说明生成时依据，不自动提高评级。

#### B. Codex 守门报告固定 Schema

Codex 守门报告必须尽量输出以下结构，便于后续机器检索、批量审计和追责：

```json
{
  "Codex_Gatekeeping_Report": {
    "module_name": "",
    "source_files_checked": [],
    "external_draft_checked": "",
    "local_source_pass_completed": false,
    "bidirectional_matrix_completed": false,
    "image_audit_completed": "yes | no | not_applicable | partial",
    "reading_coverage_basis": "",
    "current_grade": "",
    "max_grade_after_repair": "",
    "verdict": "archive_ready | codex_reinforce | return_to_kimi | source_missing | pre_gate_only",
    "blocking_issues": [],
    "repair_actions": [],
    "positive_flywheel_candidate": false,
    "standard_version": {}
  }
}
```

使用规则：

- `archive_ready` 只能在本地证据、双向核对和必要图像审计达到当前标准时使用。
- `pre_gate_only` 表示只做了文件验证、抽取统计、关键词/目录/缩略图预审，不能当正式守门。
- `positive_flywheel_candidate = true` 时，必须说明外部初稿或本次处理有哪些做法高于现行规则，并向用户申请是否制度化。

#### C. 问题清单 / 修复队列统一字段

所有 P0/P1/P2 问题清单、补强队列、批量审计问题表应尽量使用以下字段：

```json
{
  "repair_issue": {
    "module_name": "",
    "severity": "P0 | P1 | P2",
    "category": "source_chain | gatekeeping | index | aigc_interface | modern_layer | density | style_gate | file_output | versioning",
    "issue": "",
    "evidence": "",
    "required_fix": "",
    "owner": "Codex | Kimi | user | external_source",
    "status": "open | fixed | deferred | wont_fix",
    "verification_method": "",
    "standard_version": {}
  }
}
```

分级规则：

- `P0`：正式归档不成立，例如路径断链、HTML/PDF 不可用、Manifest 主入口冲突、严重混源、缺关键源书证据。
- `P1`：模块可用但不达新标准，例如 AIGC 接口薄、现代适配层混乱、正文密度不足、缺 Codex 双向守门。
- `P2`：轻量修复，例如命名、标签、备注、sidecar 链接、格式锚点或局部索引备注。

### 3.6 文件夹纯净度审计流程

定期检查：

- 根目录是否只有 `AGENT.md` 和 `目录说明.md`。
- `html/PDF/正式归档_已升级` 是否只放正式成品。

发现不确定内容，先放 `06_旧件隔离_待确认`，不直接删除。

## 4. 外部模型稿守门流程

当用户提供 Kimi、DeepSeek、Gemini、Claude 或其他模型的蒸馏稿时，Codex 是守门员。处理顺序如下：

1. 定位源书：在 `书籍资料\` 中找到对应 EPUB/PDF/DOCX/MOBI。
2. 定位外部稿：在 `kimi初步蒸馏\` 中找到对应初稿、交接包或补强报告。
3. 来源审计：确认文件格式、大小、是否可抽取文本、是否扫描版、是否需要 OCR、是否有图像教学价值。
4. Codex 自读原书：可读文本必须完整处理；大书可分块，但不能只抽样。若只能抽样，必须标为预审并写明评级上限。
5. 阅读披露核对：外部稿声称全文阅读时，必须用本地证据校验章节覆盖、正文长度、目录、关键词分布、页图/图像审计和结构单元覆盖。
6. 双向核对矩阵：逐项记录：
   - `source_to_draft_omissions`：原书中重要但外部稿漏掉/弱化/误读的内容。
   - `draft_to_source_unsupported_claims`：外部稿中无本地证据、证据不足、错误归因或与原书冲突的内容。
   - `image_or_diagram_cross_check`：图像/图表描述是否被本地视觉审计支持。
   - `modern_layer_cross_check`：现代补强是否正确标注为 `modern_supplement`，是否误称为原书内容。
7. 反伪蒸馏检查：判断是否只有概念摘要，还是具备智能体可执行层。
8. 评级判断：给出当前评级、可冲评级、硬上限原因。
9. 补强方案：若不足，明确缺什么、由 Codex 补还是需要用户/Kimi 重蒸。
10. 用户确认：除非用户明确要求直接做，否则先输出守门结论，不要跳过询问直接正式归档。
11. 正式生成：确认后再生成 HTML/PDF、更新索引、记录替换关系。
12. 第二步自检：正式输出后再次检查评级、索引、链接、文件位置、旧版处理。

外部模型的“我已全文阅读”不能直接采信。若和本地证据冲突，本地证据优先。
外部稿很详细也不能替代 Codex 自读与双向核对；详细只是可参考，不是可信证明。

### 4.1 Kimi/外部模型初步蒸馏稿最低交付标准

为了让 Codex 能高质量守门，Kimi、Gemini、Claude、DeepSeek 等外部模型的初稿不能只写总结，也不能只说“全文已读”。合格初稿至少应包含：

1. `Source Audit`：源书路径或文件名、格式、大小、可抽取性、OCR/扫描状态、版本/译本信息。
2. `Reading Coverage`：已读结构单元清单，含章节、附录、图像/图表/表格范围；若有未读或抽样，必须列出。
3. `Evidence Map`：核心概念、案例、图像观察、方法论分别来自哪些章节或页段；不能只靠记忆转述。
4. `Knowledge Layer Map`：明确区分 `original_book`、`modern_supplement`、`related_module`、`unverified_external`、`unavailable_source`。
5. `Self-Contamination Audit`：自查是否把现代品牌、现代平台、熟悉案例、外部知识替换成了原书没有的内容；发现问题要列出并修正。
6. `Executable Draft`：触发条件、输入字段、控制参数、工作流、输出 Schema、Prompt 模板、QC、Negative Constraints、Repair Rules、Cross-Module Dispatch、Black-Box JSON 的初版。
7. `AIGC Handoff Draft`：若服务视频/广告/影视/小说改编/视觉生产，必须给 production stages、upstream inputs、video control parameters、prompt hooks、video QC、repair 和 cross-shot consistency 的初稿。
8. `Codex Handoff Notes`：列出需要 Codex 判断的评级上限、现代适配争议、图像审计缺口、AIGC 接口缺口、可能混源点和建议补强方向。

不合格初稿的处理规则：

- 只有“全文已读”但无结构覆盖证据：只可作为预审材料，不可直接正式守门。
- 只有章节摘要而无执行层：最高按方向卡处理，不能进入正式 A 级归档。
- 图像型/电影图册/设计书只读文字不审图：必须降级或退回补图像审计。
- 现代平台、AIGC、短视频、电商、软件工具内容未标 `modern_supplement`：按混源风险处理。
- 把原书没有的品牌、案例、导演风格、技术名词写成原书事实：必须进入双向核对矩阵修复。

### 4.2 外部模型质量反馈与正向连带飞轮迭代规则

外部模型初稿不是只被动等待 Codex 守门的材料，也可以成为本项目规则升级的探针。Codex 必须判断外部初稿相对当前 `AGENT.md` 与 `agent-book-distillation` 的标准位置，并触发不同处理。

质量状态统一使用：

- `below_current_standard`：低于现行规则。必须打回、要求重蒸，或由 Codex 明确补强；不得硬凑正式归档。
- `meets_current_standard`：达到现行规则。进入正常 Codex 本地证据守门、双向核对、正式归档流程。
- `exceeds_current_standard`：明显高于现行规则。Codex 必须第一时间提醒用户，并提炼可复用的新标准候选。

当初稿低于现行标准：

1. Codex 输出打回原因和缺失字段。
2. 明确缺口属于来源证据、阅读覆盖、图像审计、知识分层、执行层、AIGC handoff、QC/repair、索引链中的哪一类。
3. 不把低质量做法写入规则，不把缺陷制度化。

当初稿达到现行标准：

1. Codex 正常执行本地源书核验与双向核对。
2. 如正式归档成功，可记录为合格样例，但不自动升级全局规则。

当初稿高于现行标准：

1. Codex 必须向用户指出：该初稿包含高于当前规则的新实践。
2. 提取可复用机制，例如更细的 Source Audit、更强的图像审计、更好的污染自检、更实用的 AIGC 控制面板、更精确的 QC/repair、更好的分层字段或索引链。
3. 判断该机制是否适合制度化：是否跨书可复用、是否不污染原书证据、是否能提升智能体可执行性、是否会增加过高成本。
4. 向用户申请是否写入 `AGENT.md`、`agent-book-distillation`，或两者都写。
5. 用户确认后再同步本地、项目和 GitHub，并记录为规则版本升级。

这会形成正向连带飞轮：

```text
高质量外部初稿
  → Codex 本地证据守门与差距识别
  → 可复用规则提炼
  → 用户确认
  → AGENT.md / Skill 制度化
  → 后续 Kimi 初稿标准提高
  → Codex 守门与正式归档标准同步提高
  → 历史模块进入补强队列，而不是被旧标准永久冻结
```

连带升级规则：

- 外部初稿标准提高后，Codex 守门标准必须同步提高；不能让低门槛守门稀释高质量初稿。
- Codex 守门标准提高后，正式 HTML/PDF 归档标准必须同步提高；不能只在守门报告里写得好，正式模块正文仍停留在旧颗粒度。
- 正式归档标准提高后，最终自检与批量审计标准必须同步提高；不能让旧字段扫描冒充新标准通过。
- 新标准不自动否定历史模块，但会生成 `legacy_reinforcement_queue`。历史模块按优先级补强，不因过去合格而永久豁免。
- 评级必须分开：新规则可提高“现代可执行性”和“AIGC 生产可用性”，但不能自动提高原书证据评级。

## 5. 每本书第一步分析必须包含

第一步分析不生成正式 HTML/PDF，除非用户明确要求。必须输出：

- 源文件路径。
- 文件格式、大小、是否加密、是否扫描版、是否可抽取文本。
- 本次是否完整阅读全文；若没有，说明读了哪些、没读哪些、为什么。
- 阅读覆盖证据：章节、目录、字符数、页数、图片数、OCR 状态等。
- 本书适合蒸馏成什么模块。
- 对智能体的能力增量。
- 预计评级：蒸馏后可达 A / A- / B+ / B / C / D。
- 硬上限：若因扫描版、缺页、OCR 噪音、外部稿不足导致不能 A，必须说明。
- 准备蒸馏的模块清单。
- 是否建议进入正式蒸馏。

## 6. 正式蒸馏必须包含的执行层

正式归档文件必须尽量包含以下结构：

1. Source Audit：来源审计。
2. Reading Coverage Disclosure：阅读覆盖披露。
3. Module Identity：模块身份、职责、适用边界。
4. Agent Capability Delta：智能体能力增量。
5. Trigger Conditions：触发条件。
6. Required Inputs：输入字段。
7. Control Parameters：控制参数，优先 sliders / enums / bools / ratios。
8. Core Methodology：核心方法论。
9. Executable Modules：可执行子模块。
10. Workflow：步骤化工作流。
11. Output Schema：输出结构。
12. Prompt Templates：可直接复用的提示模板。
13. Quality Checker：质量检查器。
14. Negative Constraints：负向约束。
15. Repair Rules：修复规则。
16. Cross-Module Dispatch：跨模块调用链。
17. Black-Box JSON：黑盒调用 JSON。
18. Search Tags：检索标签、别名、Obsidian 反链。
19. Final Rating：最终评级和可用性判断。
20. AIGC Video Production Interface：视频生产阶段、内容类型、上游输入、视听控制参数、下游交付、生成接口。
21. Video Quality And Repair：跨镜头一致性、视听结果检查、失败分类和修复规则。

缺少触发、输入、控制参数、工作流、输出、质检、修复中的关键项时，不得轻易给 A。
对本项目中直接或间接服务视频生产的模块，如果缺少 AIGC Video Production Interface，最高评级为 B+ / A-；若只能阅读、不能进入生产调用链，最高评级为 B。

### 6.1 AIGC 视频接口的最低颗粒度

| 模块类型 | 必须转化的生产接口 |
|---|---|
| 叙事/编剧 | 场景目标、冲突、节拍、人物状态、信息揭示、场景进出点、改编约束 |
| 人物/对白/表演 | 角色目标、潜台词、动作动词、情绪转折、表演强度、口吻、视线与反应 |
| 镜头/导演/摄影 | 景别、机位、焦距、摄影机运动、调度、轴线、视线、空间连续性 |
| 美术/世界观/风格 | 场景资产、建筑、材质、服化道、时代、文化、天气、视觉母题与禁用元素 |
| 光线/色彩 | 光源动机、方向、软硬、对比、曝光、色温、色彩关系、镜头匹配和连续性 |
| 剪辑/声音 | 剪辑点、节奏、时长、转场、声音视点、环境声、音乐功能和声画关系 |
| 广告/品牌/传播 | 受众、欲望、品牌人格、传播目标、Hook、脚本功能、证明、CTA、渠道版本 |
| 小说/文学 | 可改编场景、人物关系状态、叙事视角、关键意象、世界规则、原作边界 |
| 理论/研究 | 决策透镜、诊断问题、证据要求、偏差检查、适用与不适用条件 |

不得为了填表让每本书承担全部视频环节。模块只输出自己负责的控制，并通过 Cross-Module Dispatch 把其他任务交给对应模块。

### 6.2 历史模块的 AIGC 视频补强规则

此前已经蒸馏、但未明确面向 AIGC 视频生产的模块，在全部书籍主体蒸馏完成后执行统一补强。

补强原则：

1. **先审计，后补强**：扫描正式升级 HTML/PDF 和 Manifest，判断是否已有视频生产接口，不按文件名猜测。
2. **增量优先**：来源审计、核心方法和执行层可靠时，不重新全文蒸馏；追加 AIGC 视频接口、Prompt Hooks、质量检查和调度关系。
3. **分级处理**：
   - `V0 无接口`：只有摘要或通用智能体结构，需要重点补强或重蒸。
   - `V1 可路由`：能说明生产阶段和上下游，但缺具体参数。
   - `V2 可执行`：具备视频输入、控制、输出、Prompt 和检查器。
   - `V3 可生产`：进一步具备跨镜头一致性、失败修复、跨模块 SOP 和可验证交付。
4. **按领域批次处理**：叙事 → 人物对白 → 镜头导演 → 美术风格 → 光色 → 剪辑声音 → 广告品牌 → 小说改编，避免逐本随机补丁。
5. **不污染原书**：AIGC 视频转换属于工程层或 `modern_supplement`；必须与 `original_book` 分离。
6. **保留主入口**：能增量补强时更新现有正式模块和 Manifest，不制造同一本书多个并列主版本。
7. **重新评级**：书籍证据评级与视频生产可用度分别记录；视频补强不能抬高原书阅读证据评级。
8. **建立补强队列**：至少记录模块名、原评级、视频接口等级、缺失字段、补强方式、优先级、上下游模块和完成状态。

历史模块补强完成的验收条件：

- 明确 `production_stages` 和 `supported_content_types`。
- 至少一个可进入视频生产的结构化输出或下游交接包。
- 至少一个生成/分析 Prompt Hook。
- 至少一个视频结果质量检查器。
- 至少一组失败修复规则。
- 涉及多镜头时包含跨镜头一致性约束。
- Manifest 可按视频任务、生产阶段和能力关键词检索到该模块。

### 6.3 AIGC 外搭知识库最终验收标准

本项目的正式归档不是“人能读懂的资料库”，而是“智能体可外搭调用的知识库”。最终验收时必须确认每个模块至少回答以下问题：

1. 何时调用：触发条件是否具体，能否和相邻模块区分。
2. 向谁要什么：Required Inputs 是否明确上游输入、用户问题、素材状态或项目约束。
3. 控制什么：Control Parameters 是否能转成可观察结果，而不是形容词堆。
4. 怎么执行：Workflow / Executable Modules 是否能让智能体按步骤运行。
5. 交付什么：Output Schema 是否结构化，能被下游模块继续消费。
6. 怎么生成：Prompt Templates / generation_prompt_hooks 是否专属本模块职责。
7. 怎么检查：Quality Checker / video_quality_checks 是否能判定失败，而非只写“质量要高”。
8. 怎么修：Repair Rules / video_repair_rules 是否给出可执行修复动作。
9. 怎么分层：原书证据、现代适配、相关模块、未验证外部知识是否分开。
10. 怎么调度：Cross-Module Dispatch 是否说明本模块不负责的任务交给谁。

AIGC 接口最低等级：

- `V3_production_ready`：核心生产模块必须达到；具备生产阶段、输入、控制参数、结构化 handoff、prompt hooks、QC、repair、连续性规则。
- `V2_executable`：理论、诊断、参考模块可接受；能输出决策、诊断问题、修复建议或路由，而不是直接生成镜头。
- `V3_when_routed`：本身不直接生产，但和专业模块组合后可形成 V3 生产链。
- `not_directly_applicable`：只允许用于确实不负责生产控制的模块，且必须写明上游参考职责和路由去向。

正式模块不得用以下写法冒充合格：

- “可用于 AIGC 视频”“可提升创作力”但没有输入、参数、输出、QC、repair。
- 只有 `Machine_Consistency_Field_Index` 或字段目录，没有正文实质内容。
- 把现代平台规则、AIGC 工具参数、当代案例写成原书观点。
- 为了安全或合规把导演风格、类型冲突、暴力/恐怖/性/政治/讽刺等影视手法一刀切删除；应改为创作意图、可观察参数、虚构/历史/伦理边界、现代适配层和风险门。

## 7. ABCD 评级标准

- A：全文/核心全文证据充分，执行层完整，可直接供后期智能体调用；有质量检查器、修复规则、跨模块链接、BlackBox JSON。
- A-：主体已可用，但存在轻微限制，如附录未全读、图像未全部参数化、外部补强来源非原书、少量参数仍需后续扩展。
- B+：可用版。方法论成立，执行层大体完整，但受 OCR、缺页、图片未审计、来源不完整或参数不够精确影响。
- B：能作为参考模块，但智能体调用仍需人工解释或二次加工。
- C：必须重蒸。通常缺来源审计、完整阅读披露、触发条件、输入字段、执行流程、评级或能力增量。
- D：不建议进入最终库。多为薄摘要、概念摘抄、伪蒸馏或无法验证来源。

评级必须说明“是否可供后期智能体调用”。不能只说好不好看。

## 8. 图像型、扫描型和 OCR 规则

遇到插图多、扫描版、构图/机位/打光/设计类书籍时，不能只读文字。

必须判断图片类型：

- 装饰图：只提供氛围，不作为核心规则。
- 案例图：用于风格识别和案例佐证。
- 教学示范图：必须转化为参数、规则或 Prompt 模板。
- 扫描噪音图：记录 OCR 风险和硬上限。

如果 OCR 或图片审计不足，必须降低评级或标为待补强。

## 9. 原书证据层与现代适配层分离规则

旧书的方法论可以适配现代媒介、平台和 AIGC 工作流，但不得把现代常识、外部模型补写或其他书籍知识冒充为原书内容。智能体知道当前时间，不等于它能自动判断每条规则的来源、年代、适用媒介和证据强度。

当出现以下任一情况时，正式分析、补强稿和归档模块必须启用来源分层：

- 原书媒介、软件、技术、渠道或操作环境明显过时。
- 需要把原书方法迁移到短视频、电商、社交媒体、AIGC、现代软件或当前平台。
- Kimi、Gemini、Claude 等外部模型加入了原书未出现的现代知识。
- 原书规则在今天需要合规、平台、技术或伦理修正。
- 本地正文缺页、OCR 不完整或只能由外部资料补足部分信息。
- 同一模块同时包含原书方法和 Codex 的工程化扩展。

至少区分以下两层：

1. `original_book`：原书证据层。只收录可由本地全文、完整 OCR、页码/章节证据或可靠版本校验支持的内容。
2. `modern_supplement`：现代适配层。收录面向当前媒介、平台、技术和工作流的补强内容，必须注明不归因于原书。

必要时可增加：

- `related_module`：由知识库中其他已归档模块负责的知识，不在本书中重复发明。
- `unverified_external`：仅有外部摘要或模型建议、尚未得到本地证据支持的候选内容；不得进入自治执行主路径。
- `unavailable_source`：原书缺页、纯黑页、损坏图像或未读范围内的内容；不得用常识自动补齐。

建议为重要规则、清单、参数和 Prompt 字段附加：

```json
{
  "knowledge_layer": "original_book | modern_supplement | related_module | unverified_external | unavailable_source",
  "source_status": "verified_local | partial_local | external_verified | unavailable",
  "attributed_to_book": true,
  "applicable_period": "original_context | current",
  "applicable_channels": [],
  "requires_current_validation": false
}
```

执行规则：

- 原书措辞即使显得老式，也不能因“不像现代标准类别”而被删除或替换；可以另加现代语义解释。
- 现代标题公式、平台术语、软件参数、短视频 Hook、CTA、A/B 测试、合规规则等可以补充，但必须放入 `modern_supplement`。
- 优先通过 Cross-Module Dispatch 调用已有现代模块，避免每本旧书重复加入同一套现代知识。
- 现代适配不能掩盖缺页、未读、OCR 噪音或原书证据不足，也不能提高原书阅读覆盖评级。
- 若现代补强与原书冲突，必须并列说明适用条件，不得静默覆盖原书。
- Quality Checker 必须检查是否发生来源错归、年代错配或把补强内容伪称为原书内容。
- Manifest 中的模块说明应能看出其主要职责是原书方法、现代适配器，还是两层组合模块。

不必机械分层的情况：

- 模块只使用原书方法，没有加入现代补强。
- 只是忠实改写表达方式，没有改变规则、适用条件或归因。
- 现代能力已由其他模块负责，且通过跨模块调度即可完成。

评级规则：

- 原书忠实度与现代可执行性分别检查。
- 现代补强再强，也不能补偿本地全文缺失或把原书证据评级抬高。
- 存在未标注知识混源、整组清单替换或错误归因时，不得评 A；若会污染智能体自治执行，最高 B，修复后再重评。

### 9.1 创作风险门不是一刀切禁用规则

在电影、恐怖、犯罪、战争、心理、历史创伤、家庭暴力、儿童恐惧、性别压迫、殖民历史、身体恐怖等创作型模块中，`Gate`、`Safety Gate`、`Blocker`、`Constraint` 不得默认理解为“禁止创作”或“削弱艺术冲突”。它们的默认含义是：

- **来源分层**：区分原书证据、影片个案、外部模型推导、现代 AIGC 转译、平台/版权/伦理约束。
- **创作路由**：判断该内容应交给镜头、表演、声音、剪辑、美术、安全 QC、历史语境或改编边界模块处理。
- **强度控制**：保留冲突、恐怖、暴力、压迫和黑暗表达，但标明表现强度、视角、镜头距离、是否镜外、是否后果化、是否象征化。
- **方法提取**：学习影视手法和叙事功能，而不是复制原片名场面、演员肖像、书内图像、具体乐曲或受保护构图。
- **质量修复**：当 AIGC 输出退化为血浆、鬼脸、猎奇、廉价 jump scare、创伤装饰或阴谋论事实化时，将其修复为更高级的叙事、空间、表演、声音和历史结构。

执行规则：

1. 黑暗内容、暴力冲突、恐怖手法、压迫关系和道德不适感本身不是降级原因。若它们来自本地原书/影片证据，并能服务叙事、角色、空间、声音、历史或美学功能，应保留并转化为可执行创作参数。
2. 只有在以下情况才使用“禁止/阻断”：真实人物可识别肖像未授权复刻、受版权保护画面/音乐精确复制、儿童或弱势主体的刺激性受害生成、操作性伤害步骤、把真实创伤当装饰、把外部阴谋论当事实、或把现代补强伪称为原书内容。
3. “风格调用”可以保留。用户明确要求某导演、类型、年代或影片气质时，应把风格名转成可观察参数，如镜头时值、机位、运动、构图、材质、声音功能、剪辑节奏、表演强度和空间关系；不得仅因风格名敏感而删除。
4. 对经典影视手法，优先写成 `allowed_use`、`intensity_control`、`representation_mode`、`repair_rule`，不要只写 `blocked`。
5. `Negative Constraints` 应区分“艺术上禁止的坏输出”和“安全/版权/来源上禁止的输出”。例如禁止“廉价鬼脸化”是质量约束；禁止“复刻某张演员脸”是肖像/版权约束；两者不能混为一谈。
6. 正式归档中的安全门命名应尽量使用“Gate / Router / Intensity Control / Representation Mode / Context Gate”，避免让后续智能体误以为所有风险内容都应删除。

建议字段：

```json
{
  "creative_risk_gate": {
    "default_policy": "route_and_control_not_blanket_ban",
    "allowed_use": [],
    "representation_mode": "direct | restrained | implied | offscreen | aftermath | symbolic | blocked",
    "intensity_level": "low | medium | high | extreme_blocked",
    "narrative_function": "",
    "source_layer": "original_book | case_derived_transfer | modern_supplement | safety_constraint",
    "must_not_degrade_into": [],
    "repair_rule": ""
  }
}
```

## 10. 旧版处理规则

若新版本已达到更高评级：

1. 不直接混放新旧版。
2. 旧版可删除、移动到 `旧版_已替换`，或备份到旧版记录区。
3. 必须写入替换关系：旧文件、新文件、替换原因、日期、评级变化。
4. 若旧版仍有特殊价值，保留并标注“历史参考，不作为主调用模块”。

如果旧版只是 C/D 级摘要且新版已 A/A-/B+，通常不再作为最终库保留。
